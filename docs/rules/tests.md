# Rules: Tests (tests/)

Apply these rules when modifying or generating test files in `tests/`.

## Do not modify test files manually
Tests in `tests/` are **generated by the AI test agent** on each PR. Do not edit them
directly. If test behaviour needs to change, update the source code or test prompts instead.

## Fixtures (tests/conftest.py)
Two shared fixtures are always available:
- `client` — Flask test client with `TESTING=True` and `SECRET_KEY` set.
- `mock_anthropic` — patches `app.client` so no real API calls are made.

Always use these fixtures via `@pytest.fixture` dependency injection.

## Mocking rules
- **Never make real Anthropic API calls in unit or functional tests.**
- Patch at `app.client` — not at the `anthropic.Anthropic` class level.
- Use `MagicMock()` for the response object; set `.content[0].text` for the quote.
- Example:
  ```python
  with patch("app.client") as mock_client:
      mock_client.messages.create.return_value = MagicMock(
          content=[MagicMock(text="Stay focused.")]
      )
  ```

## Coverage requirements
- Minimum 70% line coverage (enforced in CI).
- Every route (`GET /`, `POST /`, `GET /health`, `GET /ping`, `GET /version`) must have at least one test.
- Test the error path: when `app.client.messages.create` raises `Exception`.
- Test empty/whitespace-only form submission.

## Test naming
- `test_unit.py` — unit tests using `unittest.mock`.
- `test_functional.py` — functional tests using Flask test client.
- Test function names: `test_<what>_<condition>` e.g. `test_post_returns_quote_on_success`.

## Assertions
- Every test function must contain at least one `assert`.
- Use `assert response.status_code == 200` not `assert response.status_code`.
- Check HTML content with `assert b"<text>" in response.data`.
